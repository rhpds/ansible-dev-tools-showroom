—— The Challenge

Automation Content Without a Workflow Is a Liability

No standard structure. Every team scaffolds collections differently. Onboarding takes weeks instead of minutes.
No testing discipline. Playbooks ship untested. Failures surface in production, not in development.
No supply chain trust. Content flows from repo to production with no verification of integrity or authorship.
—— The Insight

One Curated Toolchain Covers the Entire Lifecycle

—— Phase 1: Create

Scaffold Collections in Seconds, Not Hours

ansible-creator

Generate complete collection scaffolds from the CLI or VS Code with a single command. Consistent structure every time — galaxy.yml, plugins, tests, molecule scenarios, all pre-wired.

# From the CLI $ ansible-creator init collection \ mynamespace.mycollection /path # Or click "Collection project" in VS Code

ade

Manage isolated virtual environments, install collections with full dependency tracking, and resolve Python and system requirements automatically.

# Install collection in editable mode $ ade install -e . # View dependency tree $ ade tree -v
—— Phase 1: Create

Write Modules, Lint on Save, Run from the IDE

Add custom modules and plugins to your collection, then validate and execute without leaving VS Code.

Custom modules — Write Python modules inside your collection's plugins/modules/ directory. Full argument spec, check mode support, and FQCN resolution out of the box.
ansible-lint --fix — Enable auto-fix in VS Code settings. Every save corrects formatting, enforces naming conventions, and flags issues inline.
ansible-navigator TUI — Run playbooks in a terminal UI that lets you drill into plays, tasks, and output interactively.
—— Phase 2: Test

Validate at Every Layer Before Production

Molecule

Integration testing with ephemeral containers. Runs the full lifecycle: create, converge, verify, destroy. Tests roles across multiple OSes and platforms.

Integration Multi-OS

pytest-ansible

Unit and functional testing for individual modules, filters, and plugins. Fast, isolated, CI-friendly. Uses pytest fixtures for Ansible execution.

Unit Tests Fast

tox-ansible

Orchestrates all tests with a 5-line config. Auto-discovers lint, unit, sanity, and molecule environments. Cross-version matrix testing.

Orchestration Matrix

—— Phase 2: Test

Five Lines of Config. Every Test Discovered.

tox.ini

[tox] envlist = lint, unit skip_missing_interpreters = True requires = tox-ansible [ansible] ansible_core_versions = 2.15 2.16

Auto-discovered environments

$ tox list lint ansible-lint py311-ansible2.16-unit pytest py311-ansible2.16-molecule- integration_hello_world molecule scenario

Convention over configuration. The plugin scans your collection and creates test environments automatically.

—— Phase 3: Deploy

Package Everything Into Portable Execution Environments

ansible-builder creates container images with your collections, Python dependencies, and system packages — ready for Ansible Automation Platform.

execution- environment.yml Define
→
ansible-builder create Generate
→
ansible-builder build -t my-ee Build
→
Push to Registry Ship
# Build and tag the execution environment $ ansible-builder build -t mynamespace/mycollection-ee:latest -vvv # Verify collections inside the image $ podman run mycollection-ee:latest ansible-galaxy collection list
—— Phase 3: Deploy

Sign Content for Supply Chain Trust

ansible-sign uses GPG signatures to ensure content integrity from development to production. Every file gets checksummed, the manifest gets signed, and AAP verifies before execution.

# Sign the project $ ansible-sign project gpg-sign . GPG signing successful! # Verify integrity $ ansible-sign project gpg-verify . GPG signature verification succeeded!

Trust Chain

Developer signs with GPG key
↓
Push signed project to Git
↓
AAP verifies signature at every sync
